# 项目结构问题分析报告

## 当前结构存在的问题

- 根目录长期平铺多个源码包，随着能力增长，模块职责边界越来越模糊
- `notebooks/` 同时承载 `.ipynb` 与共享 Python 模块，实验资产与工程代码混放
- 训练脚本长期兼容“包导入”和“裸脚本直跑”两种模式，导致存在 `sys.path` 注入和 fallback import
- 配置资产、脚本入口、源码实现没有形成统一分层，路径变更影响面大
- 根级 `README.md` 与部分历史文档不能准确反映实际架构，团队认知成本高
- 辅助工具域与主业务链路并列平铺，缺少统一主包约束

## 对照主流 Python/ML 工程规范的差距

- 缺少统一源码根，未采用主流的 `src/<package>` 结构
- 缺少标准脚本入口层，运行方式过于依赖源码文件物理位置
- Notebook 共享逻辑没有完全下沉到独立包层
- 配置资产未完全独立到专门目录
- 文档没有围绕“主包 + 入口 + 资源规范”形成统一说明

## 重构目标

- 建立统一主包 `src/unsloth_finetune/`
- 以 `core/data/training/notebooking/tools` 完成清晰分层
- 将 `scripts/` 作为标准运行入口
- 将 `configs/` 与 `requirements/` 独立出来，降低路径耦合
- 保留根级兼容层，降低一次性迁移风险

## 迁移原则

- 优先保证运行链路稳定，再逐步清理兼容层
- 优先调整内部标准导入到 `unsloth_finetune.*`
- notebook 保持可用，但共享逻辑全部以下沉包为准
- 配置与文档同步更新，避免“代码已迁移、说明仍旧路径”
